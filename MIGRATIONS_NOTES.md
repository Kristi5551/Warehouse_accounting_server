# Миграции и начальные данные (Flyway)

## Легаси: `V8__insert_initial_data.sql`

- **Не редактировать** после того, как миграция применена на любом окружении: Flyway сверяет checksum; изменение файла сломает `validate` / деплой.
- Рассчитана на **последовательное применение на пустой схеме** после V1–V7: в ней используются **предполагаемые числовые id** (`category_id` 1–3, `product_id` / строки остатков 1–4). На «грязной» БД с другим порядком вставок возможны **конфликты или неверные связи** — это известное ограничение легаси-сидов.
- **V9** и последующие версии не должны повторять хрупкие id из V8; там, где возможно, используются **естественные ключи** (имя склада, `article`, email в коде).

## Что добавлять дальше

1. **Новые изменения схемы** — только новые файлы `V{N}__*.sql` (монотонно увеличивать `N`).
2. **Новые демо/сид-данные** — предпочтительно:
   - идемпотентные выражения: `INSERT … ON CONFLICT … DO NOTHING` / `WHERE NOT EXISTS`;
   - ссылки по **имени, article, email**, подзапросам `SELECT id FROM … WHERE name = …`, а не литералам вида `id = 1`.
3. **Критические строки без Flyway** (админ, страховка склада) — только в **`InitialDataSeed.kt`**, уже по естественным ключам (email, имя склада).

## Связка с Kotlin

- **`InitialDataSeed`** не дублирует наполнение категорий/товаров из V8 на успешном стандартном прогоне; он дополняет учётку администратора и при необходимости создаёт склад «Основной склад», если его ещё нет.
- Если нужен полностью переносимый сид без привязки к id — добавляйте либо новую идемпотентную миграцию по правилам выше, либо расширяйте `InitialDataSeed` / **отдельный сервис** с теми же правилами.
